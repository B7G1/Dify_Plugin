多数据库查询插件设计方案（含 DM / KingbaseES 扩展）

1 设计需求

这个是一份插件设计文档，我们想要从 0 开始设计一个新的 Dify 数据库查询插件，而且是根据现在已经有了的现有插件 junjiem-db_query_0.0.11-offline.difypkg 作为结构参考。导入本地插件之后，我们把它加入到新节点，看其中的用户输入和下滑选择栏，我们发现它已经体现了 Dify 前端下拉栏、Tool 参数、数据库驱动映射、requirements.txt 和 wheels/ 离线依赖组织方式。

所以我们的目标是设计一个多数据库查询插件，而且还要确保基础能力覆盖参考包中已经出现的数据库类型，最后在同一结构中扩展 DM 和 KingbaseES。

  • MySQL
  • Oracle
  • Oracle 11g
  • PostgreSQL
  • Microsoft SQL Server
  • DM（达梦数据库）
  • KingbaseES（人大金仓数据库）

同时满足：

  • 完全离线部署
  • 本地依赖管理
  • 插件打包导出
  • 导入 Dify 后可作为 Workflow 节点使用

1.1 1. 项目目标

为什么需要这个插件？因为我们希望在 Dify Workflow 中直接查询数据库。这样的话，用户就不需要在工作流外手动查库、复制结果、再粘贴给后续节点了，直接在一个 Tool 节点中完成查询就可以了。

这个插件要解决四个具体问题：

问题 插件要提供的能力
Workflow 需要读取业务数据 提供 SQL 查询节点
数据库类型不止一种 支持 MySQL、Oracle、Oracle11g、PostgreSQL、SQL Server，并扩展 DM、KingbaseES
部署环境不能访问公网 所有依赖本地化，支持离线安装
结果要被后续节点使用 返回结构化 JSON，同时支持 Markdown 展示

我们大概描述这个插件的理想回路就是这样的：

连接数据库 -> 执行只读 SQL -> 返回查询结果

所以这个项目的目标是：

设计一个可离线交付的 Dify 多数据库查询 Tool Plugin，让 Workflow 能以节点形式选择数据库类型、执行只读 SQL，并返回可继续流转的查询结果。

1.2 2. 总体设计思路

我们打算设计一个 Dify Tool Plugin。用户在 Dify 中通过下拉栏选择数据库类型，填写连接信息，在 Workflow 节点中输入 SQL，插件调用对应数据库驱动执行查询，然后最后再把结果返回给 Dify就可以了。

总体架构图如下：

设计思路分为四层：

层级 设计内容
Dify 接入层 让插件以 Tool 的方式出现在 Workflow 中
参数配置层 管理数据库类型、连接信息、SQL 和输出格式
数据库适配层 按数据库类型选择驱动和连接方式
离线交付层 将源码、依赖、驱动和资源打包为 .difypkg

这样Dify只需要理解一个 Tool然后把数据库差异留在插件内部处理就可以了。这样的好处就是会人性化很多。

1.3 3. 模块分类

插件应该拆成哪些部分？

我们看了官方的文档，然后发现这个按职责可以拆成六类：

模块分类如下：

模块 负责什么 不负责什么
配置模块 声明插件、Provider、Tool、参数表单 不执行 SQL
SQL 执行模块 接收 SQL、校验 SQL、发起查询 不关心具体驱动安装
数据库适配模块 处理 MySQL、Oracle、Oracle11g、PostgreSQL、SQL Server、DM、KingbaseES 的连接差异 不直接暴露给用户
结果处理模块 将查询结果转为 JSON / Markdown 不改变查询语义
打包模块 管理 requirements.txt、wheels/、.difypkg 不处理业务逻辑
校验模块 验证连接、依赖、导入、查询、离线运行 不替代测试环境

这样相当于，等我们新增数据库的时候，只要主要修改数据库适配模块和依赖包就可以了，不需要重写 Tool 调用流程。

1.4 4. 插件类型定义

我们看了Dify官方对插件的三大分类，最后选了 Tool Plugin。（其他两个是Model Provider和Endpoint。因为这两个一个是用来接入模型能力的，一个是给外部调用使用然后给人家开放API的，这俩确实都不提供我们希望用到的内部功能。）

数据库查询符合 Tool 的调用方式：

需求 Tool Plugin 是否匹配
在 Workflow 中作为节点使用 匹配
在 Agent 中作为工具调用 匹配
前端展示参数表单 匹配
接收 SQL 并执行动作 匹配
返回 JSON 给后续节点 匹配
打包为 .difypkg 导入 Dify 匹配

结论：

本插件定义为 Dify Tool Plugin。

值得一提的是，插件类型配置应体现在 manifest.yaml 中。这个在我们之前的报告里有提到。

type: plugin
plugins:
  tools:
    - provider/db_query.yaml
resource:
  permission:
    tool:
      enabled: true

目录职责如下：

main.py 插件运行入口。启动 Dify Plugin SDK

_assets/ 插件图标和展示资源
.difyignore 控制哪些文件不进入最终插件包

provider 和 tool 的边界要清楚：

层级 做什么
数据库配置模块（Provider） 保存数据库地址、账号密码等信息，然后检查数据库能不能连上
SQL 查询模块（Tool） 接收用户输入的 SQL 语句，到数据库执行后返回结果

1.6 6. 数据库扩展设计

我现在想要知道怎么在现有多数据库结构上支持 DM 和 KingbaseES，最好的办法就是先定义统一输入：

参数 说明
db_type 数据库类型：mysql、oracle、oracle11g、postgresql、mssql、dm、kingbase
host 数据库地址
port 数据库端口
database 数据库名
username 用户名
password 密码，使用密文输入
connect_timeout 连接超时时间
query_sql SQL 查询语句
output_format json 或 markdown

数据库驱动映射表如下：

数据库 驱动 连接路线 适配方式
MySQL PyMySQL SQLAlchemy 常规适配
Oracle oracledb SQLAlchemy 常规适配
Oracle 11g oracledb + thick client SQLAlchemy 特殊初始化适配
PostgreSQL psycopg2 / psycopg2-binary SQLAlchemy 常规适配
Microsoft SQL Server pymssql SQLAlchemy 常规适配
DM dmPython DB-API 或专用 dialect 独立适配
KingbaseES ksycopg2 PostgreSQL 兼容路线 兼容适配

参考包中的 db_type 下拉栏已经包含了 mysql、oracle、oracle11g、postgresql、mssql。我们肯定要保留这些选项，然后在此基础上再新增 dm 和 kingbase。这样 Dify 前端看到的是一个完整的查询工具了。

我们研究了每个数据库和Dify的适配路线，发现大概是这样子：

已有数据库类型 设计处理
MySQL 保留 mysql+pymysql 路线
Oracle 保留 oracle+oracledb 路线
Oracle11g 保留 Oracle 路线，并在适配器中处理 thick client 初始化
PostgreSQL 保留 postgresql+psycopg2 路线
SQL Server 保留 mssql+pymssql 路线

这里面 DM 和 KingbaseES 是这次要求里增加的两个适配分支。为了支持这些数据库，我们要补齐驱动依赖、连接方式、异常处理、离线 wheel 和验收用例。

1.6.1 6.1 DM 扩展方案

因为达梦比较特殊，所以大概率要做独立适配。DM 使用 dmPython 作为 Python 驱动，这就告诉我们它肯定不是和 MySQL 或 PostgreSQL 轻松完全兼容的，所以做独立适配器肯定更好。

DM 适配器负责：

设计点 方案
驱动 使用 dmPython
连接串 使用 DM 驱动要求的 host、port、user、password、database 参数
SQL 执行 通过 DB-API cursor 执行
结果读取 从 cursor description 提取列名，从 rows 转成 records
异常处理 捕获 DM 驱动异常，转换成 Dify 可读错误
依赖打包 将 dmPython 对应 wheel 或安装包放入 wheels/

连接逻辑示意：

import dmPython

connection = dmPython.connect(
    user=username,
    password=password,
    server=host,
    port=port,
)

我们总结了一下DM 独立适配的原因：

原因 说明
驱动行为独立 dmPython 的连接参数、异常类型、返回值细节可能不同
离线依赖特殊 DM 驱动常需要单独准备，不能默认从公网获取
问题定位清晰 DM 连接失败时，需要返回 DM 相关错误，而不是通用 SQLAlchemy 错误

1.6.2 6.2 KingbaseES 扩展方案

我们刚刚说了DM的适配路线，那问题是 KingbaseES 为什么可以和 PostgreSQL 用一样的技术路线？这是因为 KingbaseES 和 PostgreSQL 兼容度较高，这样的话插件设计上就不用从零做一套查询链路了，我只要替换驱动和少量连接参数就可以了。

KingbaseES 适配器负责：

设计点 方案
驱动 使用 ksycopg2
技术路线 PostgreSQL 兼容模式
SQL 执行 复用 PostgreSQL 风格连接和 cursor
结果读取 与 PostgreSQL 返回结构保持一致
异常处理 将 ksycopg2 异常转换为插件统一错误
依赖打包 将 ksycopg2 wheel 放入 wheels/

我们希望连接路线是这个样子：

kingbase
  -> ksycopg2
  -> PostgreSQL-compatible connection
  -> cursor.execute(sql)
  -> records

这样设计解决的问题是：KingbaseES 不需要占用一套完全独立的实现了，但是还可以保留单独的适配器，这样就方便后续处理兼容差异了。

1.7 7. SQL 查询执行流程设计

我们这部分是插件设计技术的一部分，因为肯定技术部分会遇到一个问题：用户输入 SQL 后，插件内部该咋执行？

SQL 执行流程图如下：

有一些实现规则，我们后面设计的时候就可以参考：

规则 原因
默认只允许单条 SELECT 降低误执行写操作风险
使用只读数据库账号 从数据库权限层兜底
设置连接超时 避免连接阶段阻塞
设置查询超时 避免长 SQL 卡住插件
设置最大返回行数 避免大结果集拖垮 Workflow
返回 JSON 方便后续节点消费
返回 Markdown 方便用户直接阅读

建议返回成 JSON 结构：

{
  "columns": ["id", "name"],
  "rows": [
    {"id": 1, "name": "demo"}
  ],
  "row_count": 1,
  "truncated": false,
  "database_type": "kingbase"
}

1.9 9. 依赖本地化与插件打包方案

我们最后的目标肯定是生成 .difypkg 然后导入 Dify 成为工作流节点，但是依赖也需要好好管理。

1.9.1 9.1 requirements 固定版本

依赖版本要固定，这也就避免不同环境安装出不同结果。

示例：

--no-index --find-links=./wheels/
dify_plugin==0.2.1
SQLAlchemy==2.0.35
PyMySQL==1.1.1
oracledb==2.2.1
psycopg2-binary==2.9.10
pymssql==2.3.4
dmPython==x.y.z
ksycopg2==x.y.z
sqlparse==0.5.3
tabulate==0.9.0

上面这些就是大概找到的一些依赖版本。--no-index 的作用是禁止访问公网包仓库。--find-links=./wheels/ 的作用是从插件包内的本地依赖目录查找 wheel。

1.9.2 9.2 wheel 下载与本地依赖仓库

跟之前说的一样，只要在联网环境中提前下载依赖就可以了：

pip download -r requirements-online.txt -d wheels/

执行这个命令到本地仓库，本地依赖仓库就是插件目录下的 wheels/。它至少要包含：

依赖类型 示例
Dify SDK dify_plugin
通用 SQL 能力 SQLAlchemy、sqlparse
MySQL 驱动 PyMySQL
Oracle 驱动 oracledb
PostgreSQL 驱动 psycopg2-binary
SQL Server 驱动 pymssql
DM 驱动 dmPython
KingbaseES 驱动 ksycopg2
输出格式化 tabulate

1.9.3 9.3 difypkg 打包

打包前要检查：

检查项 要求
manifest.yaml type: plugin，plugins.tools 指向 Provider
Provider YAML 包含连接凭据配置和 Tool 引用
Tool YAML 参数与 Tool Python 代码一致
requirements.txt 固定版本，指向 wheels/
wheels/ 所有依赖存在，平台匹配
.difyignore 排除测试缓存、临时文件、真实凭据

打包命令比如说写成这样：

dify plugin package ./multi_db_query

Windows 本地命令可写为：

.\dify-plugin.exe plugin package .\multi_db_query -o .\dist

1.9.4 9.4 导入 Dify

根据所演示的前端中新建了智能体，然后本地导入步骤：

1. 打开 Dify 插件管理页面
2. 选择本地文件导入
3. 上传生成的 .difypkg
4. 确认插件权限
5. 完成安装

1.9.5 9.5 成为工作流节点

插件成为 Workflow 节点的条件：

条件 说明
插件类型是 Tool Plugin Dify 才能在工具节点中选择
Tool YAML 参数完整 Workflow 节点才能渲染输入项
插件已安装到 Workspace 工作流中才能引用
Provider 凭据通过校验 节点运行时才能连接数据库
Tool 返回 JSON 后续节点才能读取查询结果

使用方式：

Workflow
  -> 添加 Tool 节点
  -> 选择数据库查询插件
  -> 选择 sql_query 工具
  -> 输入 query_sql
  -> 运行节点
  -> 后续节点读取 rows / columns / row_count
