## Provider 层结论

- `provider/db_query.yaml` 没有定义 `credentials_for_provider`。
- `provider/db_query.py` 的 `_validate_credentials()` 是空实现：`pass`。
- 数据库连接参数不在 Provider 级配置，而是在 Tool 参数中由用户/LLM 每次调用时传入。
- 这意味着 Dify 插件安装或 provider 配置阶段不会验证数据库连通性。
- Provider 与 Tool 的关联通过 `provider/db_query.yaml` 中的 `tools: - tools/sql_query.yaml` 完成。

## Tool 层结论

`tools/sql_query.yaml` 定义 Tool：`sql_query`。

主要参数：

- `db_type`：select，支持 mysql/oracle/oracle11g/postgresql/mssql
- `db_host`：string
- `db_port`：number
- `db_username`：string
- `db_password`：secret-input
- `db_name`：string
- `db_properties`：string
- `query_sql`：string
- `output_format`：select，支持 markdown/json

`tools/sql_query.py` 的 `_invoke()` 主要流程：

1. 读取并校验数据库类型、host、username、password、SQL 等参数。
2. 使用 `sqlparse.parse(query_sql)` 解析 SQL。
3. 限制只能传入一条语句。
4. 使用 `statement.get_type() != 'SELECT'` 限制只能执行 SELECT。
5. 创建 `DbUtil(...)`。
6. 调用 `db.run_query(query_sql)`。
7. 根据 `output_format` 返回 JSON 或 Markdown 表格。

## db_util 层结论

`tools/db_util.py` 是核心数据库访问层。

关键实现：

- 使用 SQLAlchemy `create_engine()` 创建连接引擎。
- 使用 pandas `read_sql_query()` 执行 SELECT。
- 使用 URL 编码处理 username/password/host/database。
- 支持 `properties` 追加到 connection string query 参数。
- 对 pandas 结果做清洗：
  - `NaN` 填充为空字符串
  - `Timestamp` 转为 `%Y-%m-%d %H:%M:%S`
  - `datetime.date` 转为 `%Y-%m-%d`
  - `UUID` 转字符串
  - 整数型 float 转 int
- 返回 `list[dict]`。

## 风险发现

| 风险 | 等级 | 说明 |
|---|---:|---|
| Provider 不校验 credentials | 中 | 安装/配置阶段不会验证数据库参数，错误会延迟到 Tool 调用时暴露 |
| 数据库连接参数全部作为 Tool 参数 | 中 | Agent 场景下 LLM 可参与填充连接参数，容易误填或泄露 |
| `db_password` 是 `secret-input` | 低/正向 | 密码字段使用 secret-input，是正确做法 |
| SQL 限制为单条 SELECT | 低/正向 | 阻止多语句和非 SELECT，降低写操作风险 |
| SELECT 安全边界仍有限 | 中 | 仍可能执行高成本 SELECT、敏感表 SELECT、函数调用等 |
| `db_properties` 直接拼接到 URL query | 中 | 如果由 LLM/用户自由输入，可能传入危险连接参数 |
| logging 输出完整 URL | 高 | `get_url()` 中 `logging.info(f"url: {url}")` 可能记录数据库用户名、密码 |
| Oracle11g thick client 依赖外部环境 | 中 | 需要 Dify plugin_daemon 容器挂载 instant client 与 LD_LIBRARY_PATH |
| offline wheel 平台受限 | 中 | wheel 多为 Linux x86_64/cp312，离线包对运行平台有要求 |

结论：静态分析完成。
