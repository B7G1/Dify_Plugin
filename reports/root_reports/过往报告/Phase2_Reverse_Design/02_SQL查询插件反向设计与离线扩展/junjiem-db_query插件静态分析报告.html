junjiem-db_query 离线插件包静态分析报告

1 junjiem-db_query 离线插件包静态分析报告

本报告基于 junjiem-db_query_0.0.11-offline.difypkg 的静态解包结果编写。分析过程未安装插件、未运行插件代码、未修改原始 .difypkg 文件，仅阅读解包后的配置、源码、依赖声明与离线 wheel 文件。

解包路径为 E:\Dify_Plugin\analysis\junjiem-db_query_0.0.11。该插件是一个 Dify Tool Plugin，核心能力是通过 Tool 参数接收数据库连接信息与 SQL，使用 SQLAlchemy 与数据库驱动执行查询，并把结果以 Markdown 表格或 JSON 返回给 Dify Workflow / Agent。

1.1.3 1.3 manifest

manifest.yaml 是 Dify Plugin 的根配置文件。它定义插件身份、运行时、资源限制、权限、Provider 入口与 Marketplace 展示信息。Dify 或 plugin-daemon 加载插件包时，会先读取该文件来判断插件如何启动、有哪些 provider、需要哪些资源。

version: 0.0.11
type: plugin
author: junjiem
name: db_query
label:
  en_US: DB Query
  zh_Hans: 数据库查询
created_at: 2025-06-17T21:43:31.911109+08:00
resource:
  memory: 1048576
plugins:
  tools:
    - provider/db_query.yaml
meta:
  version: 0.0.11
  arch:
    - amd64
    - arm64

作用： 声明插件身份、版本、类型、资源、Provider 入口与目标架构。
输入： 插件包根目录中的 YAML 元数据。
输出： plugin-daemon 可识别的插件加载配置。
关联文件： provider/db_query.yaml、README.md、PRIVACY.md、_assets/。
在插件生命周期中的位置： 插件安装、加载、运行前的第一层配置入口。

manifest 中的 resource.memory: 1048576 表示插件运行时资源声明。plugins.tools 指向 provider/db_query.yaml，这使 Dify 能找到该插件暴露的 Tool Provider。meta.arch 声明 amd64 与 arm64，但离线 wheels 主要体现 Linux x86_64 / cp312 生态，实际离线可用性仍需要以 wheel 文件平台标签为准。

1.1.4 1.4 main.py

main.py 是插件运行进程的入口，通常保持很薄，只负责读取环境变量、启动 SDK runtime，然后把具体能力委托给 Provider 和 Tool。

import os
from dify_plugin import Plugin, DifyPluginEnv

plugin = Plugin(DifyPluginEnv(MAX_REQUEST_TIMEOUT=120))

if __name__ == '__main__':
    plugin.run()

作用： 启动 Dify Plugin SDK runtime。
输入： plugin-daemon 注入的运行环境与请求。
输出： 一个可与 plugin-daemon 通信的插件进程。
关联文件： manifest.yaml、Provider YAML、Tool Python 文件。
在插件生命周期中的位置： 插件被加载后，运行期进程启动的入口。

该文件没有业务逻辑。仿写自己的数据库插件时，建议保持类似结构，把数据库连接、参数校验、SQL 执行都放在 Tool 或独立工具层，避免 main.py 变成业务容器。

1.2 2 Provider

1.2.1 2.1 Provider YAML

provider/db_query.yaml 声明 Provider 的身份、显示名称、图标、关联 Tool，以及 Provider Python 源文件。它是 manifest 与具体 Tool 之间的桥梁。

identity:
  name: db_query
  author: junjiem
  label:
    en_US: DB Query
    zh_Hans: 数据库查询
tools:
  - tools/sql_query.yaml
extra:
  python:
    source: provider/db_query.py

作用： 声明 Provider，并把 tools/sql_query.yaml 挂到该 Provider 下。
输入： Provider 身份、展示信息、Tool YAML 路径。
输出： Dify 可展示和加载的 Provider 定义。
关联文件： provider/db_query.py、tools/sql_query.yaml。
在插件生命周期中的位置： 插件加载后，Dify 构建工具列表和 Provider 面板时使用。

该 Provider YAML 没有声明 credentials_for_provider。因此数据库连接参数不是在 Provider 层统一配置，而是作为 Tool 参数传入。这样做更灵活，但不适合生产级数据库插件，因为连接信息会进入每次 Tool 调用链路，也更难集中校验与复用。

1.2.2 2.2 Provider Python

provider/db_query.py 继承 Dify Plugin SDK 的 ToolProvider，提供 _validate_credentials 钩子。当前插件中该函数为空实现。

from dify_plugin import ToolProvider

class DbQueryProvider(ToolProvider):
    def _validate_credentials(self, credentials: dict) -> None:
        pass

作用： 提供 Provider 层 credential 校验入口。
输入： Provider 级 credentials 字典。
输出： 校验通过则无返回，失败时应抛出异常。
关联文件： provider/db_query.yaml、tools/db_util.py。
在插件生命周期中的位置： 用户配置 Provider 凭据并点击验证时触发。

当前实现没有实际校验逻辑，也不会主动连接数据库测试凭据。若我们开发自己的数据库插件，建议把数据库连接信息放在 Provider credentials 中，并在这里调用类似 DbUtil.test_sql() 的逻辑执行轻量只读连通性检查。

1.3 3 Tool

1.3.1 3.1 Tool YAML

tools/sql_query.yaml 定义 Dify Workflow / Agent 中用户能看到的工具名称、描述、参数类型、默认值和输出格式。

identity:
  name: sql_query
parameters:
  - name: db_type
    type: select
    required: true
    options:
      - value: mysql
      - value: oracle
      - value: oracle11g
      - value: postgresql
      - value: mssql
  - name: db_password
    type: secret-input
    required: true
  - name: query_sql
    type: string
    required: true

作用： 声明 Tool 入参和表单展示方式。
输入： 数据库类型、连接信息、SQL、输出格式。
输出： Dify 前端可渲染的工具表单与 Agent 可调用的参数 schema。
关联文件： tools/sql_query.py。
在插件生命周期中的位置： 用户在 Workflow / Agent 中配置和调用 Tool 时使用。

该 Tool 暴露的主要字段包括 db_type、db_host、db_port、db_username、db_password、db_name、db_properties、query_sql 和 output_format。其中 db_password 使用 secret-input，这是正确方向；但数据库 host、username、password 全部作为 Tool 参数出现，生产环境更适合迁移到 Provider credentials。

1.3.2 3.2 Tool Python

tools/sql_query.py 是真正的 Tool 执行入口。它读取参数、做基础校验、用 sqlparse 限制单条 SELECT，再调用 DbUtil.run_query() 执行查询，最后根据 output_format 生成 JSON 或 Markdown。

class SqlQueryTool(Tool):
    def _invoke(self, tool_parameters: dict[str, Any]) -> Generator[ToolInvokeMessage]:
        db_type = tool_parameters.get("db_type", "")
        db_host = tool_parameters.get("db_host", "")
        db_username = tool_parameters.get("db_username", "")
        db_password = tool_parameters.get("db_password", "")
        query_sql = tool_parameters.get("query_sql", "")

        statements = sqlparse.parse(query_sql)
        if len(statements) != 1 or statements[0].get_type() != "SELECT":
            raise ValueError("Only one SELECT statement is allowed")

        db = DbUtil(db_type, db_host, db_port, db_username, db_password, db_name, db_properties)
        records = db.run_query(query_sql)

作用： 执行 Tool 调用，完成参数校验、SQL 类型限制与数据库查询。
输入： Dify 传入的 tool_parameters。
输出： ToolInvokeMessage，可为 JSON 或文本消息。
关联文件： tools/sql_query.yaml、tools/db_util.py。
在插件生命周期中的位置： 用户在 Workflow / Agent 中触发 Tool 时执行。

if output_format == "json":
    yield self.create_json_message({"records": records})
else:
    yield self.create_text_message(
        tabulate.tabulate(records, headers="keys", tablefmt="github", floatfmt="")
    )

作用： 根据输出格式封装查询结果。
输入： DbUtil.run_query() 返回的 records。
输出： JSON 消息或 Markdown 表格文本。
关联文件： tabulate、Dify Plugin SDK message API。
在插件生命周期中的位置： Tool 执行末尾，向 Dify 返回结果。

当前 SQL 安全限制的优点是：只允许单条语句，并且语句类型必须是 SELECT。不足是：SELECT 仍可能读取敏感表、执行高成本查询，且没有 schema 白名单、表白名单、最大行数、超时控制或字段脱敏策略。

1.4 4 Database Layer

1.4.1 4.1 db_util

tools/db_util.py 是数据库适配层。它把插件参数转换为 SQLAlchemy connection string，创建 engine，并使用 pandas 执行查询和转换结果。

def get_driver_name(self):
    if self.db_type == "mysql":
        return "mysql+pymysql"
    if self.db_type == "oracle" or self.db_type == "oracle11g":
        return "oracle+oracledb"
    if self.db_type == "postgresql":
        return "postgresql+psycopg2"
    if self.db_type == "mssql":
        return "mssql+pymssql"

作用： 根据 db_type 选择 SQLAlchemy dialect 与 driver。
输入： mysql、oracle、oracle11g、postgresql、mssql。
输出： SQLAlchemy driver name。
关联文件： requirements.txt、tools/sql_query.yaml。
在插件生命周期中的位置： 创建数据库 engine 前执行。

def get_url(self):
    username = parse.quote_plus(self.db_username)
    password = parse.quote_plus(self.db_password)
    host = parse.quote_plus(self.db_host)
    database = parse.quote_plus(self.db_name)
    url = f"{self.get_driver_name()}://{username}:{password}@{host}"
    if self.db_port:
        url += f":{self.db_port}"
    if database:
        url += f"/{database}"
    return url

作用： 构造 SQLAlchemy connection URL。
输入： 数据库类型、host、port、username、password、database。
输出： SQLAlchemy 可识别的连接字符串。
关联文件： tools/sql_query.py、数据库 driver。
在插件生命周期中的位置： Tool 创建 DbUtil 实例并准备连接数据库时执行。

def run_query(self, query_sql: str):
    query_sql = query_sql.replace("%", "%%")
    df = pd.read_sql_query(
        sql=query_sql,
        con=self.engine,
        parse_dates="%Y-%m-%d %H:%M:%S",
    )
    df = df.fillna("")
    return self.to_records(df)

作用： 执行 SQL 查询，并把 DataFrame 转为 records。
输入： 单条 SELECT SQL。
输出： list[dict] 风格的记录列表。
关联文件： pandas、SQLAlchemy engine、tools/sql_query.py。
在插件生命周期中的位置： SQL 通过安全检查后，实际访问数据库时执行。

支持数据库与 driver 对应关系如下：

数据库 SQLAlchemy driver name Python driver
MySQL mysql+pymysql PyMySQL
PostgreSQL postgresql+psycopg2 psycopg2-binary
Oracle oracle+oracledb oracledb
Oracle 11g oracle+oracledb oracledb，并额外依赖 thick client 初始化
SQL Server mssql+pymssql pymssql

pandas.read_sql_query() 参与了 SQL 执行与结果读取，tabulate 则在 Tool 层把 records 转成 GitHub 风格 Markdown 表格。db_util.py 对空值使用 fillna("")，并处理 Timestamp、date、UUID、整数型浮点数等 JSON 友好转换。

1.5.2 5.2 offline package

该包名包含 offline，并且 requirements.txt 使用 --no-index --find-links=./wheels/，说明安装依赖时优先从插件包内的 wheels/ 读取，不访问 PyPI。wheels/ 目录中包含 54 个 wheel 文件，平台标签大量出现 cp312、manylinux、x86_64。

这说明插件运行时大概率面向 Python 3.12 的 Linux x86_64 环境。虽然 manifest 声明了 amd64 与 arm64，但离线安装能否覆盖 arm64 仍取决于实际 wheel 文件是否包含对应架构。

1.7 7 风险分析

1.7.1 7.1 已有安全设计

插件已经做了两项有价值的限制：一是使用 secret-input 接收数据库密码，避免密码作为普通文本字段展示；二是使用 sqlparse 限制只允许单条 SELECT 语句，降低误执行写操作或多语句注入的风险。

1.7.2 7.2 主要风险

风险 说明 建议
Provider 校验为空 _validate_credentials() 直接 pass，不能在配置阶段验证数据库连接 在 Provider 层增加连通性测试
凭据作为 Tool 参数 host、username、password 每次随 Tool 调用传入 改为 Provider credentials
SELECT 限制不充分 SELECT 仍可能读取敏感表或执行高成本查询 增加 schema 白名单、表白名单、最大行数和超时
db_properties 直接拼接 连接属性可影响 driver 行为 限制允许的连接参数集合
日志泄露风险 如果记录完整 connection URL，可能包含密码 日志中必须脱敏 password
离线 wheels 平台限制 wheels 体现 cp312 / Linux x86_64 特征 打包时按目标 Python ABI 和 CPU 架构准备 wheels
Oracle 11g 环境依赖 可能需要 thick client 在文档与运行环境中明确依赖

1.8 8 对本项目启发

1.8.1 8.1 可以复用的结构

适合复用的是整体分层结构：manifest.yaml 负责插件声明，Provider YAML 负责挂载 Tool，Provider Python 负责凭据校验，Tool YAML 负责工具参数，Tool Python 负责调用入口，db_util.py 负责数据库连接与查询。这个结构清晰，适合我们后续开发本地化数据库 Tool Plugin。

SQLAlchemy 统一 MySQL、PostgreSQL 等数据库连接的思路也值得保留。输出层提供 JSON 和 Markdown 两种格式，适合同时服务 Workflow 自动处理和 Agent 可读展示。

1.8.2 8.2 可以参考但不能照搬的代码

不能照搬空的 Provider credential 校验，也不建议照搬“所有连接信息都作为 Tool 参数”的设计。生产插件应把数据库连接配置放到 Provider credentials，并在 Tool 调用中只暴露 query_sql、output_format、可选 limit 等运行参数。

db_util.py 中构造 URL 和执行查询的思路可以参考，但需要补充 password 脱敏日志、连接超时、查询超时、最大返回行数、错误分类和更明确的异常消息。

1.8.3 8.3 支持 Dameng DM 的扩展点

如果要支持 Dameng DM，至少需要改三处：

  1. 在 tools/sql_query.yaml 的 db_type options 中新增 dm 或 dameng。
  2. 在 tools/db_util.py 的 driver 映射中新增 Dameng 对应的 dialect / driver。
  3. 在 requirements.txt 和 wheels/ 中加入 Dameng Python driver 或 SQLAlchemy dialect 相关依赖。

如果 Dameng driver 不能通过 SQLAlchemy dialect 稳定接入，则需要在 db_util.py 中增加一条 native DB-API 执行路径，不强行复用 SQLAlchemy URL。

1.8.4 8.4 driver 依赖设计

MySQL 可以继续使用 PyMySQL，PostgreSQL 可以使用 psycopg2-binary 或按部署要求改为源码版 psycopg2。Dameng DM 的 driver 要特别确认 Python 版本、系统平台、授权和 wheel 可用性。离线打包时，必须确保 wheel 的 Python ABI 与插件 runtime 对齐，例如 Python 3.12 对应 cp312。

1.8.5 8.5 credentials 设计

建议在 Provider 层设计以下 credentials：

字段 类型 建议
db_type select MySQL / PostgreSQL / Dameng DM
db_host text-input 必填
db_port number 按数据库提供默认值
db_name text-input 必填或按数据库类型决定
db_username text-input 必填
db_password secret-input 必填
connect_timeout number 提供默认值
allowed_schemas text-input 可选白名单
max_rows number 默认限制返回行数

Tool 参数则应尽量收敛为 query_sql、output_format、可选 limit。这样既方便 Workflow / Agent 使用，也能减少敏感信息暴露。

1.8.6 8.6 SQL 安全限制

本项目建议至少实现以下限制：

限制 目的
单语句 防止多语句注入
SELECT-only 避免写操作
禁止危险关键字 防止绕过和高危操作
schema / table 白名单 控制可访问范围
最大行数 防止大结果集拖垮插件
查询超时 防止长查询阻塞
只读数据库账号 从数据库权限层兜底
日志脱敏 避免泄露密码和敏感 SQL

1.8.7 8.7 输出格式设计

建议保留 JSON 和 Markdown 双输出。JSON 应包含稳定 schema，例如：

{
  "columns": ["id", "name"],
  "rows": [{"id": 1, "name": "demo"}],
  "row_count": 1,
  "truncated": false
}

Markdown 适合 Agent 直接展示，但不应作为自动化流程的唯一输出。对于大结果集，应返回截断说明、总行数或建议用户增加过滤条件。

1.8.8 8.8 offline 打包注意事项

离线插件打包时要注意 Python runtime、操作系统、CPU 架构和 wheel ABI 的一致性。若 manifest 声明 Python 3.12，则 wheels 应包含 cp312。如果目标环境是 Linux amd64，就需要 manylinux x86_64 wheel；如果要支持 arm64，还需要准备对应 arm64 wheel。

离线包中不应包含真实数据库账号、测试密码或内网连接信息。打包前应检查 README.md、示例配置、日志和测试文件，避免泄露敏感数据。

1.9 9 结论

junjiem-db_query 是一个结构清晰、适合学习的 Dify Tool Plugin 示例。它展示了从 manifest 到 Provider、Tool、数据库工具层、离线依赖打包的完整链路，也提供了多数据库查询、SQLAlchemy 统一连接、Markdown / JSON 双输出等可复用思路。

对我们自己的本地化数据库插件而言，最值得仿写的是项目结构、Tool 调用链路、SQLAlchemy 适配层和离线 wheels 打包方式。最需要改进的是 Provider 层凭据设计、连接校验、SQL 安全策略、日志脱敏、最大行数与超时控制，以及对 Dameng DM driver 的独立适配。

当前插件适合作为“参考实现”，不建议作为生产代码直接照搬。后续开发应以 MySQL / PostgreSQL 为第一阶段目标，在结构稳定后再扩展 Dameng DM，并把安全限制、凭据管理和离线打包验证作为核心验收项。
