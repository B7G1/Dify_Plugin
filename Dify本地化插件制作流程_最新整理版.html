最小 Tool Plugin 示例结构

一个最小 Tool Plugin 通常包含以下文件结构：

my_tool_plugin/
├─ manifest.yaml
├─ main.py
├─ provider/
│  ├─ my_tool.yaml
│  └─ my_tool.py
└─ tools/
   ├─ hello.yaml
   └─ hello.py

各文件职责：

文件 职责
manifest.yaml 声明插件元信息、运行器、权限、provider 路径
main.py 启动 dify_plugin SDK runtime
provider/my_tool.yaml 声明 provider identity、credentials、tools 列表、provider Python 入口
provider/my_tool.py 实现 ToolProvider，通常用于校验 credentials
tools/hello.yaml 声明具体工具 identity、description、parameters、Python 入口
tools/hello.py 实现 Tool._invoke()，返回 text/json/file 等 ToolInvokeMessage

最小调用关系：

manifest.yaml
  -> plugins.tools
    -> provider/my_tool.yaml
      -> tools/hello.yaml
        -> tools/hello.py

2. Tool Plugin 生命周期

Tool Plugin 生命周期可以拆成开发期、调试期、安装期、运行期、发布期。

运行时生命周期：

  1. Dify 根据插件安装信息识别 provider 与 tool。
  2. 用户配置 provider credentials。
  3. daemon 调用 provider 的 _validate_credentials() 校验凭据。
  4. App/Workflow/Agent 发起工具调用。
  5. daemon 路由到插件运行时。
  6. SDK 加载对应 Tool 类。
  7. Tool 的 _invoke(tool_parameters) 以 generator 形式返回消息。
  8. Dify 将 text/json/files 等结果暴露给用户或 Workflow 后续节点。

3. manifest.yaml 全字段说明

官方 manifest 是插件最基础的声明文件，格式错误会导致解析或打包失败。

顶层字段

字段 类型 必填 说明
version version/string 是 插件自身版本，例如 0.0.1
type string 是 当前主要为 plugin
author string 是 作者/组织名；发布和安装校验中很关键
name string 常见必填 插件唯一名，通常小写、无空格
label object 是 多语言展示名
description object 常见 多语言描述，官方插件普遍使用
created_at RFC3339 是 创建时间，Marketplace 要求不能晚于当前时间
icon string 是 图标路径，媒体文件通常放 _assets
resource object 否/常见 资源与权限申请
plugins object 是 声明插件扩展能力入口
meta object 是 manifest 与运行时元数据
privacy string Marketplace 需要 隐私政策文件路径或 URL
verified boolean 官方插件常见 官方/市场侧标记字段
repo / homepage string 部分插件可能有 仓库或主页信息，视模板/市场规范变化
minimum_dify_version string/null 官方插件可见 最低 Dify 版本约束

6. plugin-daemon 工作机制

dify-plugin-daemon 是插件生命周期管理服务。官方 README 描述其负责三类 runtime：

Runtime 工作方式 适用场景
Local runtime daemon 启动插件子进程，通过 STDIN/STDOUT 通信 本地部署、已安装插件运行
Debug runtime daemon 监听端口，等待开发中的插件连接，TCP 双向通信 python -m main 远程调试
Serverless runtime 插件打包部署到 AWS Lambda 等，daemon 通过 HTTP 调用 SaaS/serverless

daemon 在系统中的职责：

  1. 维护 workspace 与插件安装状态。
  2. 接收 Dify API Server 的 HTTP 请求。
  3. 根据插件安装信息选择 runtime。
  4. 启动或连接插件运行时。
  5. 转发 invocation 请求。
  6. 收集插件返回消息。
  7. 管理依赖、运行目录、日志、存储、远程调试连接等。

简化机制图：

daemon 开发/部署相关点：

  • daemon 是 Go 项目。
  • 依赖数据库、Redis、对象存储等环境变量。
  • 使用 uv 管理 Python 插件依赖。
  • PYTHON_INTERPRETER_PATH 可指定 Python 解释器。
  • 官方 README 提到 daemon 社区版对横向扩展有限，企业部署场景需关注 Redis namespace、存储卷性能等。

9.1 manifest 特征

SQLite 插件 manifest.yaml 声明：

  • version: 0.0.6
  • type: plugin

  • resource.memory: 268435456

这说明 Tool Plugin 的 manifest 不直接列出每个 Tool，而是先指向 provider yaml，再由 provider yaml 列出 tool yaml。

9.2 provider yaml

provider/sqlite_plugin.yaml 包含：

identity:
  author: langgenius
  name: sqlite
  label:
    en_US: SQLite
credentials_for_provider:
  database_path:
    type: text-input
    required: true
    label:
      en_US: Database File Path
tools:
  - tools/create_table.yaml
  - tools/delete_sql.yaml
  - tools/insert_sql.yaml
  - tools/insert_json.yaml
  - tools/select_sql.yaml
  - tools/update_sql.yaml
  - tools/update_json.yaml
extra:
  python:
    source: provider/sqlite_plugin.py

要点：

  • credentials_for_provider 定义安装/配置插件时的凭据表单。
  • database_path 是普通文本输入，要求插件运行环境能访问该 SQLite 文件。
  • tools 列出该 provider 下的多个工具。
  • extra.python.source 指向 provider 逻辑代码。

9.3 provider 代码

provider/sqlite_plugin.py 的核心职责是校验凭据：

  • 检查 database_path 是否存在。
  • 检查路径是否是文件。
  • 使用 sqlite3.connect() 执行 SELECT 1。
  • 失败时抛出 ToolProviderCredentialValidationError。

这体现了 provider 层的定位：不执行具体业务工具，而是做 provider 级凭据与连通性验证。

9.4 Tool yaml

以 select_sql.yaml 为例：

  • identity.name: select_sql
  • description.human 面向用户展示
  • description.llm 面向 LLM 选择工具时理解用途
  • parameters 定义入参
  • form: llm 表示参数由 LLM 推理填充
  • extra.python.source: tools/select_sql.py

9.5 Tool 代码

select_sql.py 的核心流程：

  1. 从 tool_parameters 获取 select_sql。
  2. 要求 SQL 非空。
  3. 要求语句以 SELECT 开头。
  4. 从 self.runtime.credentials 获取 database_path。
  5. 检查数据库文件存在。
  6. 执行 SQL。
  7. 读取 columns、rows。
  8. 组装标准 JSON：
    • query
    • table
    • columns
    • data
    • row_count
    • status
  9. 同时返回文本消息和 JSON 消息。

返回模式：

yield self.create_text_message(text_output)
yield self.create_json_message(sql_result)

这与官方 Tool Return 规范一致：工具默认可输出 text、json、files 等变量，Workflow 后续节点可以引用。

9.6 设计评价

优点：

  • provider 与 tool 分层清晰。
  • 每个 SQL 操作独立为一个工具，便于 LLM 选择。
  • select 限制只允许 SELECT，降低误操作风险。
  • JSON 输出结构稳定，便于 Workflow 消费。
  • 使用 generator 返回多条消息，符合 SDK 模式。

风险与注意点：

  • insert_sql、update_sql、delete_sql 属于写操作，使用时需要工作区权限和提示词约束。
  • database_path 是运行环境路径，云端/容器环境中路径可达性是常见问题。
  • select_sql 只用字符串前缀判断 SELECT，复杂 SQL 安全边界有限。
  • SQLite 文件并发写入、锁等待、权限问题需要在生产环境额外评估。

10. 推荐 Tool Plugin 目录结构

my_tool_plugin/
├─ manifest.yaml
├─ main.py
├─ pyproject.toml
├─ uv.lock
├─ README.md
├─ PRIVACY.md
├─ .env.example
├─ .difyignore
├─ _assets/
│  └─ icon.svg
├─ provider/
│  ├─ my_provider.yaml
│  └─ my_provider.py
└─ tools/
   ├─ action_a.yaml
   ├─ action_a.py
   ├─ action_b.yaml
   └─ action_b.py

Provider 示例：

identity:
  author: your-github-handle
  name: sample
  label:
    en_US: Sample
credentials_for_provider:
  api_key:
    type: secret-input
    required: true
    label:
      en_US: API Key
tools:
  - tools/search.yaml
extra:
  python:
    source: provider/sample.py

Tool 示例：

identity:
  name: search
  author: your-github-handle
  label:
    en_US: Search
description:
  human:
    en_US: Search external data.
  llm: Use this tool when you need to search external data.
parameters:
  - name: query
    type: string
    required: true
    label:
      en_US: Query
    human_description:
      en_US: Search keyword.
    llm_description: Search keyword.
    form: llm
extra:
  python:
    source: tools/search.py
